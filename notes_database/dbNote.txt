## What "performance" means?
(incl. Latency vs throughput optimization and trade-offs)
###
The performance of a database has 4 factors:
*** Response time / Latency (How long it takes for a single operation/query to complete (e.g. p95 or p99 latency).),
*** Throughput / Capacity (How many operations/queries the system can handle per second (ops/sec, transactions/sec, users served)),
*** Scalability (how well the system handles growing volume or num. of users.),
*** Resource efficiency (CPU, memory, I/O, network usage).
 —---
Different databases optimize for different performance profiles:
*** Graphj: Neo4j uses index-free adjacency — relationships are stored as direct pointers. Traversing from one node to another is O(1), whereas SQL needs to perform index lookups + joins → O(log n).
Downside: Slow at “table scans” (eg. “calculate average salary of all emplyeees”) or aggregations. Scaling is more complex than document stores (though possible with clustering).
*** Document: Optimized for easy sharding (horizontal scaling). High write throughput + Reduced CPU overhead during reads/writes (BSON format, no complex transformations or joins). REASON: Documents are self-contained → most queries become simple key lookups. Sharding key design allows the database to distribute data and load easily across many nodes.
Downside: High storage overhead (data dublication), Slow write of duplicated data (user changes name == must be changed everywhere). Data fragmentation and inconsistent latency (if a doc grows beyond the memory allocation, it must be moved)
*** SQL: good for complex analytical queries + joins. Downside: Joins at very large scale.
Pros: SQL excels at predictable latency for complex joins/aggregations with proper indexes and offers strong ACID support, making it efficient for transactional workloads with good resource use when data fits in memory.
Cons: It struggles with horizontal scalability (sharding is hard), suffers from join overhead (O(log n)) at large scale, and has lower write throughput due to index maintenance, constraints, and locking.
Trade-offs:
Prioritizes query flexibility and consistency over raw speed and massive scale — denormalization helps reads but hurts writes, and vertical scaling is preferred over easy horizontal growth.
This is why we migrate to MongoDB (better sharding & single-doc speed) and Neo4j (O(1) traversals) in the project.
—---
Latency VS throughput - tradeoff:
*** Techniques that improve throughput (batching, queuing, compression, write buffering) usually increase latency.
*** Techniques that improve latency (more indexes, more RAM, denormalization, caching) often reduce maximum throughput or increase resource consumption.

## Execution plans: incl.: 1: Why are they generated by the database,
what information they show? 2: How can they help us with the design?
3: Show and example.
###
Execution Plans are generated by the database optimizer to determine the most efficient way to run a query.
They show scan types, index usage, join methods, estimated costs, and row counts.
They help database design by revealing missing indexes, expensive operations, and bad modeling choices, allowing us to improve schema, indexing strategy, and data modeling (especially during SQL → MongoDB/Neo4j migration).
----
*** MySQL: EXPLAIN SELECT * FROM orders WHERE customer_id = 123;
Output (simplified): type: ALL (Full Table Scan)
rows: 1,245,678
Extra: Using where
→ Problem: No index on customer_id
*** MongoDB: db.orders.find({ customer_id: 123 }).explain("executionStats");
Output (simplified): stage: COLLSCAN (Collection Scan)
nReturned: 2450 (number of documents returned)
executionTimeMillis: 185
→ No index → scans entire collection.
*** Neo4J: PROFILE MATCH (u:User)-[:PLACED]->(o:Order)
WHERE u.name = 'John'
RETURN o;
Output (simplified): Operator: NodeByLabelScan (scans all User nodes)
Rows: 45,230
DB Hits: High
→ Inefficient if many users.
*** Execution plans show how each database executes queries.
MySQL: Shows scan type (ALL vs ref) and index usage.
MongoDB: Shows COLLSCAN vs IXSCAN.
Neo4j: Shows NodeIndexSeek (index) vs NodeByLabelScan (every node) and relationship traversal cost.

## Index usage vs. misuse.
###
*** SQl: good practice: * Create indexes on columns frequently used in WHERE, JOIN, ORDER BY, and GROUP BY clauses.
* EXPLAIN / EXPLAIN ANALYZE to verify index usage.
*** SQL misuse: * Over-indexing: Too many indexes slow down INSERT, UPDATE, and DELETE because every write must update all relevant indexes.
* Indexes on low-cardinality columns (e.g., gender, status flags) often provide little benefit.
* Unused indexes waste space and maintenance overhead.
* Creating indexes "just in case" without analyzing query patterns.
Trade-off: Great for read-heavy OLTP workloads; costly in write-heavy scenarios.
*** MongoDB:
* indexes are crucial — without them, MongoDB performs collection scans (full scan of every document).
* Create indexes on fields used in filters (find, $match), sorts, and aggregations.
* Compound indexes are very powerful when query patterns involve multiple fields in a specific order (leftmost prefix rule).
*** MongoDB misuse:
* High write-to-read ratio collections suffer because every insert/update must update all indexes.
* Too many indexes (max 64 per collection) → high storage and write overhead.
Trade-off: Indexes bring huge read gains but are expensive on writes. Monitor with $indexStats.
*** Neo4J:
* Indexes (and constraints) on node properties that are frequently used for lookups (WHERE n.property = value).
* Especially important on label + property combinations for fast node finding at the start of graph traversals.
*** Neo4J Misuse:
* Over-indexing slows down writes (each index is a redundant copy of data).
* Relying only on indexes instead of proper graph modeling (relationships are the real strength of Neo4j — "index-free adjacency").
Trade-off: Indexes help with entry points into the graph, but the power of Neo4j lies in relationship traversal, not in indexing everything

## Concurrent Access and Locking
###
*** MySQL:
Main Approach:
Primarily pessimistic concurrency control using locking + MVCC (Multi-Version Concurrency Control) in modern systems like PostgreSQL.
* Row-level locking (most common) + Table-level locks when needed.
* Shared (S) locks for reads, Exclusive (X) locks for writes.
* Two-Phase Locking (2PL): Locks acquired as needed, held until end of transaction (commit/rollback).
* Isolation levels control behavior (READ COMMITTED, REPEATABLE READ, SERIALIZABLE).
Advantages: Strong consistency, good for financial/banking data.
Disadvantages: Can cause blocking, deadlocks, and reduced concurrency under high load.
Common Issues: Deadlocks (automatically detected and resolved by aborting one transaction), lock contention on hot rows.
*** MongoDB:
Main Approach:
Optimistic Concurrency Control at document level + multi-granularity locking at higher levels.
* Single-document operations are atomic by default (no explicit lock needed).
* Uses WiredTiger storage engine with reader-writer locks and intent locks (IS/IX) at global/database/collection levels.
* Multi-document ACID transactions (since v4.0): Use snapshot isolation + locking on modified documents.
Advantages: High write throughput and scalability. Good concurrency for most workloads.
Disadvantages: Multi-document transactions are heavier and can have higher abort rates under heavy contention. Developers often implement manual optimistic locking using version fields (_version or timestamps).
*** Neo4J:
Main Approach:
Pessimistic locking on nodes and relationships + ACID transactions.
* Locks are taken automatically on nodes/relationships when they are read and then written (or explicitly with apoc.lock.nodes).
* Write locks are held until the end of the transaction.
* Strong focus on preventing lost updates during concurrent modifications of the same node/relationship.
* Deadlocks are possible and detected (one transaction is aborted).
Advantages: Excellent for maintaining relationship consistency under concurrency.
Disadvantages: Can suffer from contention on "hot" nodes (frequently updated central nodes). Graph traversals benefit from proper modeling to minimize locking scope.

## N+1 problem in the backend.
###
The N+1 problem (also called the "N+1 query problem" or "select N+1 problem") is a common performance anti-pattern in backend applications. It occurs when the application executes one initial query to fetch a list of N parent records, and then performs N additional queries (one for each parent) to fetch related child records.
Total queries = 1 + N, which becomes very expensive as N grows (e.g., 1 + 1000 = 1001 queries).
-----------------------
Bad:
"""
authors = db.query("SELECT * FROM authors WHERE ...")  # returns 100 authors
for author in authors:
    # N queries (one per author)
    books = db.query("SELECT * FROM books WHERE author_id = ?", author.id)
    author.books = books
"""
This commonly happens with ORMs (Hibernate, SQLAlchemy, Entity Framework, Mongoose, etc.) when using lazy loading by default.
Lazy loading (default in many ORMs): Related data is loaded only when accessed.
#### questionlist performance
#####
Solution: (in SQL):
Eager loading / JOINs:SQL
"""
SELECT * FROM authors
LEFT JOIN books ON books.author_id = authors.id
"""
------------------------------
Solution: (in Mongo DB):
Embedded documents (denormalization) — best way to avoid N+1 entirely:
"""
{
  "_id": "...",
  "name": "Author",
  "books": [ {...}, {...} ]   // embedded array
}
"""
Use $lookup (aggregation pipeline) for references, but prefer embedding when possible.
Project only needed fields and use proper indexing.
-------------------------------
Solution: (in Neo4J):
Use single Cypher queries with patterns instead of multiple traversals:
"""
MATCH (a:Author)-[:WROTE]->(b:Book)
WHERE ...
RETURN a, collect(b) AS books
"""
Avoid doing N separate MATCH queries in a loop.
######
General Techniques
--- DataLoader pattern (especially important in GraphQL).
--- Caching (Redis, query caching).
--- Denormalization (common in NoSQL).
--- Pagination + careful relationship loading.
--- Profiling tools (New Relic, Datadog, EXPLAIN in SQL, MongoDB profiler, Neo4j query logs).
--------------------------------
Key Points to Remember for Exam
--- N+1 = 1 query for parents + N queries for children.
--- Most dangerous with lazy loading in ORMs and GraphQL.
--- SQL → solved with JOINs / eager fetching.
--- MongoDB → solved with embedding / careful aggregation.
--- Neo4j → solved with rich pattern matching in one query.
--- Always measure with query logs or profilers.
--- Trade-off: Eager loading / embedding can increase memory usage and cause over-fetching (waterfall vs. cartesian explosion).
---------------------------------
This anti-pattern appears in almost every migration project from SQL to NoSQL because developers often keep the same "relational thinking" (multiple small queries) instead of leveraging the strengths of the target database.




## Caching strategies
###
Caching is the technique of storing copies of frequently accessed data in a fast storage layer (usually in-memory, e.g., RAM) so that future requests for that data can be served much quicker than going to the primary data store (disk-based database).
Benefits:
--- Dramatically improves read performance (sub-millisecond responses).
--- Reduces load on the primary database (SQL, MongoDB, Neo4j).
--- Lowers latency and increases throughput/scalability.
--- Cost-effective for read-heavy workloads.
------------------------------------
Trade-offs:
--- Stale data (consistency issues).
--- Additional complexity (invalidation, eviction).
--- Memory usage and potential cache stampede/thundering herd problems.
-------------------------------------
Consistency & Invalidation
--- Cache invalidation is one of the hardest problems in computer science.
,Write-through / Write-behind handle it better.
, In Cache-Aside: manually delete or update cache entry on writes.
--- Use versioning or timestamps for optimistic invalidation.
--- Event-driven invalidation (e.g., via database change streams or CDC).
#### questionlist performance
#####
common strategies (only 2/5):
--- Cache-Aside (Lazy Loading).
HOW IT WORKS: App checks cache first. On miss: fetch from DB, store in cache, return.
PROS: Simple, flexible, app-controlled
CONS: App must handle misses & invalidation
BEST FOR: Most common, read-heavy
--- Read-Through
HOW IT WORKS: Cache sits between app & DB. On miss, cache fetches from DB automatically.
PROS: Cleaner app code
CONS: Cache provider must support it
BEST OFR: High-traffic reads.
---- Also: Write-Through, Write-Behind (Write-Back), Write-Around.
-----------------------------------
Eviction Policies (When cache is full)
--- LRU (Least Recently Used) — most common.
--- LFU (Least Frequently Used).
--- TTL (Time To Live) — expire after X seconds/minutes.
--- FIFO, Random, etc.
######
Caching in the Context of Your Exam Project (SQL → MongoDB → Neo4j)
--- In SQL: Relational model → more joins → higher benefit from caching.
--- In MongoDB: Denormalization/embedding reduces need for caching but use Redis for hot paths.
--- In Neo4j: Excellent for connected data; focus on page cache + parameterized queries + result caching for common traversals.
--- Migration tip: Introduce caching early to maintain performance parity or improvement after migration. Measure with query profilers before/after.
-----------------------------------------
Key Points to Memorize for Oral Exam
--- Cache-Aside is default choice for most apps.
--- Write-Through for strong consistency, Write-Behind for performance.
--- Invalidation is critical — stale reads are a common pitfall.
--- Always monitor cache hit ratio (aim > 80-90%).
--- Parameterized queries + proper indexing are free caching wins.
--- Caching complements, does not replace, good data modeling (especially in NoSQL).



## Optimizing read-heavy vs write-heavy workloads.
(incl. OLTP vs OLAP workloads)
###
Read-Heavy vs Write-Heavy
--- Read-heavy: Many more reads than writes (e.g., analytics dashboards, social feeds). Goal: fast queries, high throughput.
--- Write-heavy: Many inserts/updates/deletes (e.g., logging, transactions). Goal: high ingestion rate, durability.
----------------------------------
OLTP vs OLAP
--- OLTP (Online Transaction Processing): Write-heavy or balanced, many small concurrent transactions. Focus: ACID, low latency, row-level ops. Typical for SQL in your project.
--- OLAP (Online Analytical Processing): Read-heavy, complex queries on large historical data. Focus: fast aggregation, column-oriented storage.
----------------------------------
#### questionlist performance
#####
Optimization Strategies
--- SQL (OLTP-focused):
Read-heavy: Indexes, materialized views, read replicas, query caching.
, Write-heavy: Proper indexing (avoid over-indexing), partitioning, batch inserts, connection pooling.
--- MongoDB:
Read-heavy: Denormalization/embedding, indexes, sharding, read replicas, aggregation pipelines with $match early.
, Write-heavy: Bulk writes, sharded clusters, write concerns tuned, capped collections.
--- Neo4j:
Read-heavy: Page cache tuning, parameterized Cypher, indexes on frequent properties, graph projections.
, Write-heavy: Batch imports (LOAD CSV), constraints/indexes, transaction batching.
######
General Tips for Your Migration Project
--- Read-heavy: Cache aggressively (Redis), denormalize in MongoDB/Neo4j.
--- Write-heavy: Use async writes, eventual consistency where possible.
--- Measure with EXPLAIN, Mongo profiler, Neo4j query logs. Scale horizontally (sharding/replication) for both.
-------------------------------
Key Trade-off: Read optimizations (denormalization, caching) can hurt write performance — balance based on workload.

## What is it (indexes) and how does it work?
###
What Are Indexes?
--- Indexes are data structures that improve the speed of data retrieval operations (SELECT, JOIN, WHERE, ORDER BY, etc.) at the cost of extra storage and slower writes.
--- They act like a book's index or a library catalog — instead of scanning every page (full table scan), you jump directly to the relevant data.
----------------------------
How Indexes Work
--- B-Tree (most common in SQL/relational): Balanced tree structure. Allows fast lookups, range scans, and sorting in O(log N) time.
--- Stores sorted copies of column values + pointers to the actual rows.
--- Hash indexes: Exact-match only (fast equality lookups, no ranges).
--- Composite indexes: On multiple columns (order matters — leftmost prefix rule).
--------------------------
Write cost: Every INSERT/UPDATE/DELETE must update the index → slower writes.
#### questionlist indexes
#####
Indexes in Your Databases
SQL:
--- CREATE INDEX on columns used in WHERE/JOIN/ORDER BY.
--- Clustered (primary key) vs Non-clustered.
--- Use EXPLAIN to check index usage.
------------------------------
MongoDB:
--- Indexes on fields or arrays (compound, text, geospatial, TTL).
--- B-tree by default. Use explain("executionStats") to analyze.
--------------------------------
Neo4j:
--- Indexes on node/relationship properties for fast property lookups.
--- Schema indexes + full-text indexes.
--- Crucial for Cypher performance on large graphs.
-------------------------------------
Best Practices
--- Index selective (high-cardinality) columns.
--- Avoid over-indexing (hurts writes).
--- Monitor usage and drop unused indexes.
--- In migration: Identify hot queries in SQL and create equivalent indexes in MongoDB/Neo4j.
-------------------------------------------
Key Trade-off: Greatly speeds up reads, but slows writes and uses more disk/memory.
