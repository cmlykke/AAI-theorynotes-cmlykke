## What "performance" means?
(incl. Latency vs throughput optimization and trade-offs)
###
The performance of a database has 4 factors:
--- Response time / Latency (How long it takes for a single operation/query to complete (e.g. p95 or p99 latency).),
--- Throughput / Capacity (How many operations/queries the system can handle per second (ops/sec, transactions/sec, users served)),
--- Scalability (how well the system handles growing volume or num. of users.),
--- Resource efficiency (CPU, memory, I/O, network usage).
 —----------------------------
Different databases optimize for different performance profiles:
--- Graphj: Neo4j uses index-free adjacency — relationships are stored as direct pointers. Traversing from one node to another is O(1), whereas SQL needs to perform index lookups + joins → O(log n).
Downside: Slow at “table scans” (eg. “calculate average salary of all emplyeees”) or aggregations. Scaling is more complex than document stores (though possible with clustering).
--- Document: Optimized for easy sharding (horizontal scaling). High write throughput + Reduced CPU overhead during reads/writes (BSON format, no complex transformations or joins). REASON: Documents are self-contained → most queries become simple key lookups. Sharding key design allows the database to distribute data and load easily across many nodes.
Downside: High storage overhead (data dublication), Slow write of duplicated data (user changes name == must be changed everywhere). Data fragmentation and inconsistent latency (if a doc grows beyond the memory allocation, it must be moved)
--- SQL: good for complex analytical queries + joins. Downside: Joins at very large scale.
Pros: SQL excels at predictable latency for complex joins/aggregations with proper indexes and offers strong ACID support, making it efficient for transactional workloads with good resource use when data fits in memory.
Cons: It struggles with horizontal scalability (sharding is hard), suffers from join overhead (O(log n)) at large scale, and has lower write throughput due to index maintenance, constraints, and locking.
Trade-offs:
Prioritizes query flexibility and consistency over raw speed and massive scale — denormalization helps reads but hurts writes, and vertical scaling is preferred over easy horizontal growth.
This is why we migrate to MongoDB (better sharding & single-doc speed) and Neo4j (O(1) traversals) in the project.
—------------------------------
Latency VS throughput - tradeoff:
--- Techniques that improve throughput (batching, queuing, compression, write buffering) usually increase latency.
--- Techniques that improve latency (more indexes, more RAM, denormalization, caching) often reduce maximum throughput or increase resource consumption.

## Execution plans: incl.: 1: Why are they generated by the database,
what information they show? 2: How can they help us with the design?
3: Show and example.
###
Execution Plans are generated by the database optimizer to determine the most efficient way to run a query.
They show scan types, index usage, join methods, estimated costs, and row counts.
They help database design by revealing missing indexes, expensive operations, and bad modeling choices, allowing us to improve schema, indexing strategy, and data modeling (especially during SQL → MongoDB/Neo4j migration).
----
*** MySQL: EXPLAIN SELECT * FROM orders WHERE customer_id = 123;
Output (simplified): type: ALL (Full Table Scan)
rows: 1,245,678
Extra: Using where
→ Problem: No index on customer_id
-------------------------
MongoDB: db.orders.find({ customer_id: 123 }).explain("executionStats");
Output (simplified): stage: COLLSCAN (Collection Scan)
nReturned: 2450 (number of documents returned)
executionTimeMillis: 185
→ No index → scans entire collection.
--------------------------
Neo4J: PROFILE MATCH (u:User)-[:PLACED]->(o:Order)
WHERE u.name = 'John'
RETURN o;
Output (simplified): Operator: NodeByLabelScan (scans all User nodes)
Rows: 45,230
DB Hits: High
→ Inefficient if many users.
---------------------------
Execution plans show how each database executes queries.
MySQL: Shows scan type (ALL vs ref) and index usage.
MongoDB: Shows COLLSCAN vs IXSCAN.
Neo4j: Shows NodeIndexSeek (index) vs NodeByLabelScan (every node) and relationship traversal cost.

## Index usage vs. misuse.
###
--- SQl: good practice: * Create indexes on columns frequently used in WHERE, JOIN, ORDER BY, and GROUP BY clauses.
- EXPLAIN / EXPLAIN ANALYZE to verify index usage.
------------------
 SQL misuse: * Over-indexing: Too many indexes slow down INSERT, UPDATE, and DELETE because every write must update all relevant indexes.
--- Indexes on low-cardinality columns (e.g., gender, status flags) often provide little benefit.
--- Unused indexes waste space and maintenance overhead.
--- Creating indexes "just in case" without analyzing query patterns.
Trade-off: Great for read-heavy OLTP workloads; costly in write-heavy scenarios.
------------------
 MongoDB:
--- indexes are crucial — without them, MongoDB performs collection scans (full scan of every document).
--- Create indexes on fields used in filters (find, $match), sorts, and aggregations.
--- Compound indexes are very powerful when query patterns involve multiple fields in a specific order (leftmost prefix rule).
--------------------
 MongoDB misuse:
--- High write-to-read ratio collections suffer because every insert/update must update all indexes.
--- Too many indexes (max 64 per collection) → high storage and write overhead.
--- Trade-off: Indexes bring huge read gains but are expensive on writes. Monitor with $indexStats.
---------------------
 Neo4J:
--- Indexes (and constraints) on node properties that are frequently used for lookups (WHERE n.property = value).
--- Especially important on label + property combinations for fast node finding at the start of graph traversals.
----------------------
 Neo4J Misuse:
--- Over-indexing slows down writes (each index is a redundant copy of data).
--- Relying only on indexes instead of proper graph modeling (relationships are the real strength of Neo4j — "index-free adjacency").
--- Trade-off: Indexes help with entry points into the graph, but the power of Neo4j lies in relationship traversal, not in indexing everything

## Concurrent Access and Locking
###
 MySQL:
Main Approach:
Primarily pessimistic concurrency control using locking + MVCC (Multi-Version Concurrency Control) in modern systems like PostgreSQL.
--- Row-level locking (most common) + Table-level locks when needed.
--- Shared (S) locks for reads, Exclusive (X) locks for writes.
--- Two-Phase Locking (2PL): Locks acquired as needed, held until end of transaction (commit/rollback).
--- Isolation levels control behavior (READ COMMITTED, REPEATABLE READ, SERIALIZABLE).
Advantages: Strong consistency, good for financial/banking data.
Disadvantages: Can cause blocking, deadlocks, and reduced concurrency under high load.
Common Issues: Deadlocks (automatically detected and resolved by aborting one transaction), lock contention on hot rows.
----------------------------
 MongoDB:
Main Approach:
Optimistic Concurrency Control at document level + multi-granularity locking at higher levels.
--- Single-document operations are atomic by default (no explicit lock needed).
--- Uses WiredTiger storage engine with reader-writer locks and intent locks (IS/IX) at global/database/collection levels.
--- Multi-document ACID transactions (since v4.0): Use snapshot isolation + locking on modified documents.
Advantages: High write throughput and scalability. Good concurrency for most workloads.
Disadvantages: Multi-document transactions are heavier and can have higher abort rates under heavy contention. Developers often implement manual optimistic locking using version fields (_version or timestamps).
---------------------------
 Neo4J:
Main Approach:
Pessimistic locking on nodes and relationships + ACID transactions.
--- Locks are taken automatically on nodes/relationships when they are read and then written (or explicitly with apoc.lock.nodes).
--- Write locks are held until the end of the transaction.
--- Strong focus on preventing lost updates during concurrent modifications of the same node/relationship.
--- Deadlocks are possible and detected (one transaction is aborted).
Advantages: Excellent for maintaining relationship consistency under concurrency.
Disadvantages: Can suffer from contention on "hot" nodes (frequently updated central nodes). Graph traversals benefit from proper modeling to minimize locking scope.

## N+1 problem in the backend.
###
The N+1 problem (also called the "N+1 query problem" or "select N+1 problem") is a common performance anti-pattern in backend applications. It occurs when the application executes one initial query to fetch a list of N parent records, and then performs N additional queries (one for each parent) to fetch related child records.
Total queries = 1 + N, which becomes very expensive as N grows (e.g., 1 + 1000 = 1001 queries).
-----------------------
Bad:
"""
authors = db.query("SELECT * FROM authors WHERE ...")  # returns 100 authors
for author in authors:
    # N queries (one per author)
    books = db.query("SELECT * FROM books WHERE author_id = ?", author.id)
    author.books = books
"""
This commonly happens with ORMs (Hibernate, SQLAlchemy, Entity Framework, Mongoose, etc.) when using lazy loading by default.
Lazy loading (default in many ORMs): Related data is loaded only when accessed.
#### questionlist performance
#####
Solution: (in SQL):
Eager loading / JOINs:SQL
"""
SELECT * FROM authors
LEFT JOIN books ON books.author_id = authors.id
"""
------------------------------
Solution: (in Mongo DB):
Embedded documents (denormalization) — best way to avoid N+1 entirely:
"""
{
  "_id": "...",
  "name": "Author",
  "books": [ {...}, {...} ]   // embedded array
}
"""
Use $lookup (aggregation pipeline) for references, but prefer embedding when possible.
Project only needed fields and use proper indexing.
-------------------------------
Solution: (in Neo4J):
Use single Cypher queries with patterns instead of multiple traversals:
"""
MATCH (a:Author)-[:WROTE]->(b:Book)
WHERE ...
RETURN a, collect(b) AS books
"""
Avoid doing N separate MATCH queries in a loop.
######
General Techniques
--- DataLoader pattern (especially important in GraphQL).
--- Caching (Redis, query caching).
--- Denormalization (common in NoSQL).
--- Pagination + careful relationship loading.
--- Profiling tools (New Relic, Datadog, EXPLAIN in SQL, MongoDB profiler, Neo4j query logs).
--------------------------------
Key Points to Remember for Exam
--- N+1 = 1 query for parents + N queries for children.
--- Most dangerous with lazy loading in ORMs and GraphQL.
--- SQL → solved with JOINs / eager fetching.
--- MongoDB → solved with embedding / careful aggregation.
--- Neo4j → solved with rich pattern matching in one query.
--- Always measure with query logs or profilers.
--- Trade-off: Eager loading / embedding can increase memory usage and cause over-fetching (waterfall vs. cartesian explosion).
---------------------------------
This anti-pattern appears in almost every migration project from SQL to NoSQL because developers often keep the same "relational thinking" (multiple small queries) instead of leveraging the strengths of the target database.




## Caching strategies
###
Caching is the technique of storing copies of frequently accessed data in a fast storage layer (usually in-memory, e.g., RAM) so that future requests for that data can be served much quicker than going to the primary data store (disk-based database).
Benefits:
--- Dramatically improves read performance (sub-millisecond responses).
--- Reduces load on the primary database (SQL, MongoDB, Neo4j).
--- Lowers latency and increases throughput/scalability.
--- Cost-effective for read-heavy workloads.
------------------------------------
Trade-offs:
--- Stale data (consistency issues).
--- Additional complexity (invalidation, eviction).
--- Memory usage and potential cache stampede/thundering herd problems.
-------------------------------------
Consistency & Invalidation
--- Cache invalidation is one of the hardest problems in computer science.
,Write-through / Write-behind handle it better.
, In Cache-Aside: manually delete or update cache entry on writes.
--- Use versioning or timestamps for optimistic invalidation.
--- Event-driven invalidation (e.g., via database change streams or CDC).
#### questionlist performance
#####
common strategies (only 2/5):
--- Cache-Aside (Lazy Loading).
HOW IT WORKS: App checks cache first. On miss: fetch from DB, store in cache, return.
PROS: Simple, flexible, app-controlled
CONS: App must handle misses & invalidation
BEST FOR: Most common, read-heavy
--- Read-Through
HOW IT WORKS: Cache sits between app & DB. On miss, cache fetches from DB automatically.
PROS: Cleaner app code
CONS: Cache provider must support it
BEST OFR: High-traffic reads.
---- Also: Write-Through, Write-Behind (Write-Back), Write-Around.
-----------------------------------
Eviction Policies (When cache is full)
--- LRU (Least Recently Used) — most common.
--- LFU (Least Frequently Used).
--- TTL (Time To Live) — expire after X seconds/minutes.
--- FIFO, Random, etc.
######
Caching in the Context of Your Exam Project (SQL → MongoDB → Neo4j)
--- In SQL: Relational model → more joins → higher benefit from caching.
--- In MongoDB: Denormalization/embedding reduces need for caching but use Redis for hot paths.
--- In Neo4j: Excellent for connected data; focus on page cache + parameterized queries + result caching for common traversals.
--- Migration tip: Introduce caching early to maintain performance parity or improvement after migration. Measure with query profilers before/after.
-----------------------------------------
Key Points to Memorize for Oral Exam
--- Cache-Aside is default choice for most apps.
--- Write-Through for strong consistency, Write-Behind for performance.
--- Invalidation is critical — stale reads are a common pitfall.
--- Always monitor cache hit ratio (aim > 80-90%).
--- Parameterized queries + proper indexing are free caching wins.
--- Caching complements, does not replace, good data modeling (especially in NoSQL).



## Optimizing read-heavy vs write-heavy workloads.
(incl. OLTP vs OLAP workloads)
###
Read-Heavy vs Write-Heavy
--- Read-heavy: Many more reads than writes (e.g., analytics dashboards, social feeds). Goal: fast queries, high throughput.
--- Write-heavy: Many inserts/updates/deletes (e.g., logging, transactions). Goal: high ingestion rate, durability.
----------------------------------
OLTP vs OLAP
--- OLTP (Online Transaction Processing): Write-heavy or balanced, many small concurrent transactions. Focus: ACID, low latency, row-level ops. Typical for SQL in your project.
--- OLAP (Online Analytical Processing): Read-heavy, complex queries on large historical data. Focus: fast aggregation, column-oriented storage.
----------------------------------
#### questionlist performance
#####
Optimization Strategies
--- SQL (OLTP-focused):
Read-heavy: Indexes, materialized views, read replicas, query caching.
, Write-heavy: Proper indexing (avoid over-indexing), partitioning, batch inserts, connection pooling.
--- MongoDB:
Read-heavy: Denormalization/embedding, indexes, sharding, read replicas, aggregation pipelines with $match early.
, Write-heavy: Bulk writes, sharded clusters, write concerns tuned, capped collections.
--- Neo4j:
Read-heavy: Page cache tuning, parameterized Cypher, indexes on frequent properties, graph projections.
, Write-heavy: Batch imports (LOAD CSV), constraints/indexes, transaction batching.
######
General Tips for Your Migration Project
--- Read-heavy: Cache aggressively (Redis), denormalize in MongoDB/Neo4j.
--- Write-heavy: Use async writes, eventual consistency where possible.
--- Measure with EXPLAIN, Mongo profiler, Neo4j query logs. Scale horizontally (sharding/replication) for both.
-------------------------------
Key Trade-off: Read optimizations (denormalization, caching) can hurt write performance — balance based on workload.

## What is it (indexes) and how does it work?
###
What Are Indexes?
--- Indexes are data structures that improve the speed of data retrieval operations (SELECT, JOIN, WHERE, ORDER BY, etc.) at the cost of extra storage and slower writes.
--- They act like a book's index or a library catalog — instead of scanning every page (full table scan), you jump directly to the relevant data.
----------------------------
How Indexes Work
--- B-Tree (most common in SQL/relational): Balanced tree structure. Allows fast lookups, range scans, and sorting in O(log N) time.
--- Stores sorted copies of column values + pointers to the actual rows.
--- Hash indexes: Exact-match only (fast equality lookups, no ranges).
--- Composite indexes: On multiple columns (order matters — leftmost prefix rule).
--------------------------
Write cost: Every INSERT/UPDATE/DELETE must update the index → slower writes.
#### questionlist indexes
#####
Indexes in Your Databases
SQL:
--- CREATE INDEX on columns used in WHERE/JOIN/ORDER BY.
--- Clustered (primary key) vs Non-clustered.
--- Use EXPLAIN to check index usage.
------------------------------
MongoDB:
--- Indexes on fields or arrays (compound, text, geospatial, TTL).
--- B-tree by default. Use explain("executionStats") to analyze.
--------------------------------
Neo4j:
--- Indexes on node/relationship properties for fast property lookups.
--- Schema indexes + full-text indexes.
--- Crucial for Cypher performance on large graphs.
-------------------------------------
Best Practices
--- Index selective (high-cardinality) columns.
--- Avoid over-indexing (hurts writes).
--- Monitor usage and drop unused indexes.
--- In migration: Identify hot queries in SQL and create equivalent indexes in MongoDB/Neo4j.
-------------------------------------------
Key Trade-off: Greatly speeds up reads, but slows writes and uses more disk/memory.

## Pros and cons (of indexes).
###
---- General Pros: Significantly faster SELECT, WHERE, JOIN, ORDER BY, and sorting.
Help enforce uniqueness (e.g. username). Improve overall query performance on large data.
---- General Cons: Slower INSERT/UPDATE/DELETE because indexes must be updated.
Consume extra storage space. Require maintenance and can hurt performance if overused.
--------------------
MySQL:
--- Pros: Excellent B-Tree indexes for equality, ranges, and joins
(great for username, user_id, rating in our project). Primary keys and foreign keys get indexes automatically.
--- Cons: Every write operation becomes slower; extra disk space used.
---------------------
MongoDB:
--- Pros: Very flexible — indexes on any field, compound indexes (multiple fields),
and special types like text indexes. Automatic _id index.
--- Cons: High memory consumption (indexes ideally stay in RAM),
slows down write-heavy workloads noticeably.
--------------------------------
Neo4j:
--- Pros: Schema indexes speed up property lookups in Cypher queries (e.g. on username or rating).
Works well with graph traversals.
--- Cons: Still slows writes; indexes mainly help filtering,
not the relationship traversals themselves.
#### questionlist indexes
#####
Verbal Summary: Indexes are crucial because they turn slow full scans into fast lookups,
boosting read performance across MySQL, MongoDB, and Neo4j.
However, they trade off write speed and storage.
In our migration project we added indexes on frequently used fields like username after moving data from SQL to NoSQL stores.
Always evaluate query patterns before creating them.

## Give typical examples when an index will not be used
and how to fix the issue.
###
When indexes are ignored in SQL (examples):
--- Functions on column (e.g. WHERE UPPER(username) = 'USER1').
--- Leading wildcard in LIKE (LIKE '%abc').
--- OR conditions or complex expressions.
-----------------------------
SQL examples:
Example: WHERE YEAR(created_at) = 2025 (function kills index).
Fix: WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01' or add functional index.
Example: WHERE role = 'PLAYER' OR role = 'ADMIN' (OR often skips index).
Fix: Use IN ('PLAYER', 'ADMIN') or UNION.
----------------------------
MongoDB examples:
Example: find({$where: "this.rating > 1500"}) (JavaScript eval, no index).
Fix: Use native operators {rating: {$gt: 1500}}.
Example: Querying unindexed field.
Fix: Create index + ensure compound index covers sort + filter.
----------------------------
Neo4J examples:
Example: WHERE toLower(u.username) = 'user1' (function on property).
Fix: Store normalized value or use WHERE u.username = 'user1'.
Example: No index on filtered property.

## What is a transaction and how does it work?
###
Definition:
A transaction is a logical unit of work that contains one or more database operations. It must be executed completely or not at all.
ACID Properties:
--- Atomicity: All or nothing (COMMIT or ROLLBACK).
--- Consistency: Database moves from one valid state to another.
--- Isolation: Transactions don’t interfere with each other.
--- Durability: Committed changes survive crashes.
-------------------------------
Mysql example:
"""
START TRANSACTION;
UPDATE Users SET role = 'ADMIN' WHERE username = 'user1';
UPDATE Players SET rating = rating + 100 WHERE user_id = 1;
COMMIT;        -- Success: all changes saved
-- or ROLLBACK; -- Failure: all changes undone
"""
--------------------------------
--- In MongoDB:
Supports multi-document transactions (since v4.0) using startSession(), startTransaction(), commitTransaction().
--- In Neo4j:
Uses Cypher transactions or driver-level transactions with automatic rollback on error.
------------------------------------
Verbal Summary:
"A transaction groups operations so they succeed together or fail together (ACID). It ensures reliability especially during migration between our three databases."

###
Advantages:
--- Atomicity: Ensures all operations succeed or none do (no partial updates).

## Name Concurrency issues that transactions solve.
###
Transactions address problems arising when multiple users/processes
access and modify shared data simultaneously (concurrency anomalies).
------------------------------
Main issues solved by transactions (via Isolation):
--- Dirty Read
A transaction reads data written by another transaction that has not yet been committed.
If the writing transaction rolls back, the reader has seen invalid ("dirty") data.
--- Lost Update
Two transactions read the same data, both modify it, and then both write back. The first update is overwritten and lost.
--- Non-repeatable Read (Inconsistent Analysis)
A transaction reads the same row twice and gets different values because another transaction updated and committed the row in between.
--- Phantom Read
A transaction re-executes a query with the same search condition and sees new ("phantom") rows that were inserted by another transaction.
------------------------------------
Additional related problems:
Inconsistent retrievals / dirty writes
Deadlocks (transactions block each other indefinitely)
-------------------------------------
How transactions help:
They guarantee ACID properties, especially Isolation (through locking, MVCC, or timestamps).
Different isolation levels (Read Uncommitted, Read Committed, Repeatable Read, Serializable)
trade off consistency vs. performance by controlling which anomalies are prevented.
#### questionlist transactions




## Isolation levels and which anomalies they prevent
###
Isolation levels (ANSI SQL standard) control the degree of concurrency vs. consistency in transactions.
Higher levels prevent more anomalies but reduce concurrency/performance.
Isolation levels define how strictly a transaction is isolated from other concurrent transactions.
They represent a trade-off between data consistency and system performance/concurrency. The ANSI SQL standard defines four main levels:
--- Read Uncommitted is the lowest isolation level. It allows dirty reads (reading uncommitted data from other transactions),
lost updates, non-repeatable reads, and phantom reads. It offers the highest concurrency but the lowest consistency.
--- Read Committed prevents dirty reads by ensuring only committed data can be read. However,
it still permits lost updates, non-repeatable reads, and phantom reads.
This is the default isolation level in many database systems such as PostgreSQL and SQL Server.
--- Repeatable Read prevents dirty reads, lost updates, and non-repeatable reads.
A transaction is guaranteed to see the same values when reading the same row multiple times.
It still allows phantom reads (new rows appearing in range queries). This level is commonly used by MySQL with InnoDB.
--- Serializable is the highest isolation level. It prevents all major anomalies: dirty reads,
lost updates, non-repeatable reads, and phantom reads. Transactions appear to execute one after another (serially),
providing the strongest consistency but with the lowest concurrency and highest risk of deadlocks.
--- Key anomalies include: dirty read, lost update, non-repeatable read, and phantom read.
Different database systems may implement these levels slightly differently or offer additional
models like Snapshot Isolation (MVCC), which behaves similarly to Repeatable Read or better in practice.
------------------------------------
#### questionlist transactions
#####
Key Notes:
--- Dirty Read: Reading uncommitted changes.
--- Lost Update: Two transactions update the same row based on earlier read → one update is lost.
--- Non-repeatable Read: Same row read twice returns different committed values.
--- Phantom Read: Range query returns different set of rows due to inserts/deletes by another transaction.
---------------------------------
Real DBMS variations:
--- Many systems use Snapshot Isolation / MVCC (e.g. PostgreSQL) which prevents most anomalies but can still allow write skew.
--- Neo4j and MongoDB have their own concurrency models (often closer to Serializable or Snapshot).
--- Trade-off: Higher isolation → fewer anomalies, more locking, risk of deadlocks, lower throughput.





## Transaction boundaries and scope: incl:
1: start -> end of a transaction.
2: What work and resources are included?
###
. Start → End of a Transaction
--- Start: A transaction begins explicitly with BEGIN TRANSACTION / START TRANSACTION (or BEGIN in some DBMS).
It can also start implicitly when the first DML statement (SELECT, INSERT, UPDATE, DELETE) is executed and autocommit is disabled.
--- End: A transaction ends successfully with COMMIT (changes are permanently saved and visible to others) or with ROLLBACK (all changes are undone).
--- The transaction ends automatically in some cases: DDL statements (CREATE, ALTER, DROP),
certain errors, connection close, or when switching to autocommit mode.
--- A transaction is therefore a bounded sequence of operations that starts at one point and ends at COMMIT or ROLLBACK.
-------------------------------------------
2. What work and resources are included?
--- Work included: All DML statements (reads and writes) executed between the start and end of the transaction. The entire group is treated as a single atomic unit — all succeed or all fail together (Atomicity).
--- Resources:
- Locks: Row-level, page-level, or table-level locks acquired on data being read or modified.
- Memory buffers / caches: Changes are held in transaction logs and buffers until commit.
- Temporary data: Session-specific temporary tables or variables created inside the transaction.
- Indexes and constraints: Any modifications to indexes, foreign keys, or triggers fired during the transaction.
- Cursor and result sets: In some cases, cursors opened within the transaction.
--- The scope is usually limited to the current session/connection. Changes made inside a transaction are not visible to other transactions until COMMIT (depending on isolation level).
------------------------------------------
In the context of your project:
--- SQL (e.g. PostgreSQL/MySQL): Strong ACID transaction boundaries.
--- MongoDB: Multi-document transactions (since v4.0) with similar startSession / commitTransaction.
--- Neo4j: Uses Cypher BEGIN, COMMIT, ROLLBACK with ACID guarantees.
--- This ensures consistency even when migrating data between these systems.

## Performance and resource costs: incl.
Why are long-running transactions not recommended?
###
Performance and Resource Costs of Transactions
Transactions consume significant resources to guarantee ACID properties:
--- Locks: Row/table/page locks held until commit/rollback.
--- Memory: Undo/redo logs, transaction buffers, and snapshot data (MVCC).
--- I/O and CPU: Write-ahead logging (WAL), index maintenance, constraint checks.
--- Network (in distributed cases): Coordination overhead.
-----------------------------------------
Why long-running transactions are strongly discouraged:
--- Lock Contention: Locks are held for the entire duration → other transactions block or wait, reducing concurrency and overall throughput.
--- Increased Deadlock Risk: Longer lock duration raises the chance of circular waits between transactions.
--- Resource Exhaustion: Consumes more memory for logs/snapshots and can cause transaction log bloat.
--- Higher Failure Probability: Network glitches, timeouts, deadlocks, or errors force full rollback of all work done so far.
--- Poor Scalability: Blocks vacuuming/cleanup processes (e.g., in PostgreSQL) and slows down the entire system.
--- Snapshot bloat (MVCC systems): Old row versions accumulate, increasing storage and read costs.
#### questionlist transactions
#####
Best Practices:
--- Keep transactions as short as possible (do all reads first, then quick writes).
--- Avoid user interaction or heavy computation inside transactions.
--- Use smaller batches when migrating large data (as in your project).
--- In MongoDB and Neo4j the same principles apply — long transactions hurt performance and increase abort rates.





## Error handling and transaction behaviour -
compare MySQL, PostgreSQL, and SQL server.
###
MySQL (InnoDB)
--- Errors are often statement-level: A failing statement rolls back only that statement; the transaction continues.
--- Some serious errors (e.g. deadlock, duplicate key in certain cases) cause full transaction rollback.
--- autocommit=1 is default → implicit transactions.
--- Less strict: you can continue after many errors. Good for high availability, but can lead to partial updates if not handled carefully.
--------------------------------
PostgreSQL
--- Very strict: Almost any error puts the transaction into an aborted state.
--- You must issue ROLLBACK before starting a new transaction. No partial continuation.
--- All changes are rolled back on error.
--- Excellent consistency, but requires careful error handling in application code (retry logic). Uses SAVEPOINTs for partial rollback.
---------------------------------
SQL Server
--- Behaviour controlled by SET XACT_ABORT ON/OFF.
--- OFF (default): Similar to MySQL — some errors (e.g. constraint violations) roll back only the statement; transaction continues.
--- ON: Most runtime errors cause full transaction rollback (like PostgreSQL).
--- Supports TRY...CATCH blocks for sophisticated error handling inside transactions.
-----------------------------------
Summary
--- PostgreSQL: Strictest (full rollback on error).
--- MySQL: Most lenient (statement-level rollback).
--- SQL Server: Configurable via XACT_ABORT.
--- For your exam project, always check the transaction state after errors and use explicit BEGIN/COMMIT/ROLLBACK with proper try-catch or retry mechanisms.

--- Restrict IP addresses allowed to connect.
--- Regularly patch and update the database server.
--- Disable unnecessary features and services.

## Full, incremental, and differential backups.
###
1. Full Backup
A complete copy of the entire database at a specific point in time.
--- Includes all data, indexes, schemas, etc.
--- Advantages: Simple restore (one file/set), easy to understand.
--- Disadvantages: Time-consuming, large storage space, high I/O load.
--- Frequency: Usually daily or weekly.
-------------------------------------
2. Incremental Backup
Backs up only the data that has changed since the last backup (full or incremental).
--- Tracks changes using timestamps or change logs.
--- Advantages: Fastest backup, smallest storage usage.
--- Disadvantages: Restore is complex and slow (need full + all subsequent incrementals).
--- Good for frequent backups (hourly).
----------------------------------------
3. Differential Backup
Backs up all data that has changed since the last full backup.
--- Advantages: Faster than full, simpler restore than incremental (full + latest differential).
--- Disadvantages: Backup size grows over time until next full backup.
--- Balance between full and incremental.
----------------------------------------
in Your Project Databases:
--- SQL (PostgreSQL/MySQL): Use pg_dump, mysqldump for full; WAL archiving + tools like pgBackRest for incremental.
--- MongoDB: mongodump (full), oplog-based incremental, or Atlas continuous backups.
--- Neo4j: neo4j-admin dump (full) + transaction logs for point-in-time.
--- Best Practice: Combine full weekly + incremental/differential daily for optimal balance of speed, storage, and recovery time. Always test restores.

## Point-in-time recovery.
###
What is it?
Point-in-Time Recovery allows restoring a database to a specific moment in time, not just to the end of a backup.
It provides granular recovery after accidental data changes, corruption, or logical errors.
-----------------------------------------------
How it Works
--- Start with a full backup as the base.
--- Replay transaction logs (WAL - Write-Ahead Log) or change logs from the backup time up to the desired recovery point.
--- Stop replay at the exact timestamp or transaction ID.
---------------------------------------------
Benefits
--- Minimizes data loss (RPO - Recovery Point Objective).
--- Enables recovery from user mistakes (e.g., DELETE without WHERE).
--- Supports compliance and auditing requirements.
--------------------------------------------------
Implementation in Your Databases
--- SQL / RDBMS (PostgreSQL): Excellent native PITR using WAL archiving + pg_basebackup + recovery.conf / postgresql.conf. Very mature and reliable.
--- MySQL: PITR via binary logs (binlog) + mysqlbinlog tool for point-in-time restore.
--- MongoDB: Available in replica sets using oplog (operations log). Atlas provides continuous backups with PITR.
--- Neo4j: Uses transaction logs for PITR in combination with full backups (neo4j-admin dump/load).
------------------------------------------------
Best Practices
--- Enable continuous WAL / log archiving.
--- Test PITR regularly.
--- Keep logs for sufficient retention period.
--- Combine with full + incremental backups.
--- PITR is a key operational capability that demonstrates strong database management knowledge for your exam project and oral exam.

## Backup consistency (hot vs. cold backups)
###
Cold Backup (Offline Backup)
--- Database is shut down or in read-only/quiescent mode during backup.
--- Guarantees perfect consistency — no changes occur while copying files.
--- Simple: copy data files, logs, configuration.
--- Advantages: Consistent by design, easy to achieve.
--- Disadvantages: Downtime required — not suitable for 24/7 systems.
-------------------------------
Hot Backup (Online Backup)
--- Database remains fully running and accessible while backup is taken.
--- Uses mechanisms like Write-Ahead Logging (WAL), snapshotting, or buffer management to maintain consistency.
--- Advantages: No downtime, supports high-availability environments.
--- Disadvantages: More complex, higher resource usage, risk of inconsistency if not implemented correctly.
--------------------------------
Consistency Comparison
--- Cold: Always transactionally consistent (frozen state).
--- Hot: Can achieve consistent backups using:
--- Fuzzy/hot backups + redo logs (e.g., PostgreSQL, MySQL InnoDB).
--- Logical backups (pg_dump, mongodump) — usually consistent per object.
--- Storage snapshots (LVM, cloud volume snapshots).
---------------------------------
In Your Project Databases
--- PostgreSQL: Hot backups via pg_basebackup or pgBackRest (with WAL).
--- MySQL: Hot backups with mysqldump (logical) or Percona XtraBackup (physical).
--- MongoDB: Hot backups with mongodump or filesystem snapshots in replica sets.
--- Neo4j: Online backups supported.
----------------------------------
Recommendation: Prefer hot backups in production with proper tools for consistency, combined with PITR for full recoverability.

## Explain the difference between using a composite PK or
having an extra PK and the FKs references.
###
1. Composite (Compound) Primary Key
--- A primary key made of two or more columns that together uniquely identify a row.
--- Example: Junction table order_items with PK (order_id, product_id).
--- Advantages:
- Natural keys — no extra column.
- Enforces uniqueness naturally.
- Can be more intuitive for some relationships.
---------------------------------------------
--- Disadvantages:
- Harder to reference (foreign keys must include all columns).
- Performance impact on joins and indexes (wider keys).
- Difficult when adding new identifying columns later.
- Not good for high-volume tables or when keys can change.
----------------------------------
2. Surrogate (Artificial) PK + FK References
--- Add an extra auto-incrementing column (e.g., id SERIAL PRIMARY KEY) as the primary key.
--- Foreign keys reference only this single id column.
--- Advantages:
- Simple, stable, and narrow primary key (better performance for joins/indexes).
- Easy to reference from other tables (single-column FK).
- Independent of business data — values never change.
- Easier to manage schema evolution.
- Standard practice in most modern applications.
-------------------------------------------
--- Disadvantages:
- Extra column and index (minor storage overhead).
- Loses natural uniqueness enforcement (requires additional UNIQUE constraint on business columns).
--------------------------------------------
Recommendation for your project:
--- Use surrogate keys (single id PK) for most tables — especially when migrating to MongoDB and Neo4j.
--- Use composite keys sparingly (e.g., pure junction tables with no extra attributes).
--- This design choice significantly affects normalization, performance, and ease of data migration.

## When does the join table become a real entity
that needs to have its own model in the backend?
###
A join table (also called junction or associative table) starts as a simple structure to resolve a many-to-many relationship between two tables.
It becomes a real entity (deserving its own model in the backend) when:
--- It has additional attributes (payload data) beyond just the two foreign keys.
--- Example: order_items table with order_id, product_id, plus quantity, unit_price, discount, added_at.
--- It has its own identity and lifecycle independent of the related entities.
--- It participates in relationships with other tables (has its own foreign keys or is referenced by others).
--- It requires its own business logic, validation rules, or operations (e.g., status, history, approvals).
--- It has unique constraints or behaviors that go beyond the pure relationship.
-------------------------------------------
Examples
--- Pure Join Table → No extra columns: user_roles(user_id, role_id) → Usually no separate backend model.
--- Becomes Entity → enrollments(student_id, course_id, enrollment_date, grade, status) → Needs its own Enrollment model with service layer, validations, etc.
---------------------------------------------
In Backend Design:
--- Use a full Entity class (e.g., OrderItem in Java/Spring, OrderItem document in MongoDB, or node in Neo4j) when it carries attributes or behavior.
--- This improves clarity, maintainability, and makes migration to document/graph models easier.
--- Rule of Thumb: If you find yourself adding columns other than the two FKs + timestamps, promote it to a real entity with its own surrogate primary key.
This concept is very important for your SQL → MongoDB → Neo4j migration project.

## Cascading rules - what should happen with
the join table record on delete?
###
Cascading Rules for Join Tables on DELETE
In a many-to-many relationship, the join table (junction/associative table) holds foreign keys to both parent tables (e.g., user_id → users, role_id → roles).
----------------------------------
Common Foreign Key Actions on DELETE
--- CASCADE: Automatically delete the join table row when a parent row is deleted.
--- RESTRICT / NO ACTION: Prevent deletion of the parent if join records exist.
--- SET NULL: Set the FK column to NULL (rarely useful in join tables).
--- SET DEFAULT: Set to default value (rarely used).
------------------------------------
Recommended Behavior for Join Tables:
Best practice: Use ON DELETE CASCADE on both foreign keys in the join table.
Why?
--- When a User is deleted, their association records (e.g., user-role links) should also be deleted automatically.
--- When a Role is deleted, the links to that role should disappear.
--- This keeps the join table clean and avoids orphaned records.
--- It reflects the semantic meaning: the relationship ceases to exist if either participant is gone.
#### questionlist join
#####
Example SQL:
"""
SQLCREATE TABLE user_roles (
    user_id  INTEGER REFERENCES users(id) ON DELETE CASCADE,
    role_id  INTEGER REFERENCES roles(id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, role_id)
);
"""
Alternatives:
--- Use RESTRICT if you want to protect important relationships (forces explicit cleanup first).
--- Avoid SET NULL in pure join tables as it usually doesn't make sense.
---------------------------------------------
In Your Project:
--- Apply CASCADE in SQL for clean data.
--- In MongoDB: Handle manually in application code or use embedding.
--- In Neo4j: Use DETACH DELETE for relationships.
--- This ensures referential integrity without leaving dangling associations.







## When moving to Document or Graph model,
how are join tables modelled?
###
When moving from a relational database (like MySQL) to Document or Graph models, the concept of "join tables" is fundamentally re-architected.
-----------------------------------
Document Model (e.g., MongoDB)
In document databases, join tables are typically replaced by two main techniques:
--- Embedding:** Related data is nested directly within the parent document. This is ideal for one-to-many relationships where the data is frequently accessed together and doesn't change independently.
--- Referencing:** Similar to foreign keys, the ID of another document is stored within the current document. In Spring Data MongoDB, this is often implemented using annotations like `@DocumentReference` or `@DBRef`.
----------------------------------
### Graph Model (e.g., Neo4j)
In graph databases, relationships are first-class citizens.
--- Relationships (Edges):** Instead of a separate "join table" to link two entities, you define direct relationships (edges) between nodes. These edges can have their own properties, effectively replacing the need for extra columns in a SQL join table.
--- In the provided codebase, this is implemented using the `@Relationship` annotation on fields within the model class (e.g., `PlayerNode` having a `@Relationship(type = "IN_COUNTRY")` to a `CountryNode`).
--------------------------------------
Summary: Join tables disappear, replaced by structural nesting in Document databases or direct connection edges in Graph databases.

## Explain a certain query scenario for all 3 database
solutions. Start with ERD then explain
the same query for MongoDB and then for Neo4j.
###
To illustrate how the different models handle data in our project,
let’s consider the scenario of finding a player's rating and country name given their username.
---------------------------------
1. Relational Database (MySQL)
ERD Perspective: In the relational model, we have three distinct tables: Users, Players, and Countries.
--- The Players table contains foreign keys: user_id (linking to Users.id) and countries_id (linking to Countries.id).
--- The Query: To get this information, you must join the tables across these foreign keys.
"""
SELECT p.rating, c.name AS country_name
FROM Players p
JOIN Users u ON p.user_id = u.id
JOIN Countries c ON p.countries_id = c.id
WHERE u.username = 'target_username';
"""
Key Point: You rely on the database engine to perform the joins at runtime based on the defined Foreign Key constraints.
-------------------------------------
2. Document Database (MongoDB)
Structure Perspective:
--- Your PlayerDocument stores a User reference (@DocumentReference) and the country as a simple String (embedded).
--- The Query: Because you have stored the country directly in the player document as a string,
you don't need a "join" for the country. You only need to resolve the user reference.
"""
// 1. Find the User document to get the ID
const user = db.users.findOne({ "username": "target_username" });
// 2. Query the Players collection using that ID
const player = db.players.findOne({ "user": new DBRef("users", user._id) });
// Result includes: player.rating and player.country (the string)
"""
Key Point: You leverage data denormalization (embedding the country name) to avoid joins, and use application-level referencing (or $lookup in aggregation pipelines) for the User entity.
--------------------------------------
3. Graph Database (Neo4j)
Structure Perspective:
--- The data is stored as nodes (User, Player, Country) connected by edges (HAS_USER, IN_COUNTRY).
--- The Query: In Neo4j, you don't "join" tables; you traverse the path between nodes.
"""
MATCH (u:User {username: 'target_username'})-[:HAS_USER]->(p:Player)-[:IN_COUNTRY]->(c:Country)
RETURN p.rating, c.name AS country_name
"""
Key Point: This is a direct traversal. The performance is highly efficient because
it follows the physical pointers (edges) stored in the database, regardless of
how many players or users exist in the system.

## Compare the general complexity and speed / scalability
(of a certain query senario from the project)
###
1. MySQL (Relational)
--- **Complexity:** Moderate. Joins (`JOIN`) depend heavily on indexed foreign keys.
--- **Speed:** Fast on structured data with proper indexing. Execution plan complexity grows with the number of joined tables and data volume.
--- **Scalability:** Strong vertical scaling. Horizontal scaling (sharding) is complex due to maintaining referential integrity across nodes.
------------------------------------
2. MongoDB (Document)
--- **Complexity:** Low to Moderate. Requires either multiple round-trips from the application or an aggregation pipeline (`$lookup`).
--- **Speed:** Very fast for single-document retrieval. Aggregation `$lookup` can become slower than SQL joins on large datasets if not properly modeled.
--- **Scalability:** Excellent. Designed for horizontal scaling via built-in sharding, making it superior for massive, distributed datasets.

3. Neo4j (Graph)
--- **Complexity:** Lowest (Path Traversal). You simply follow pre-existing pointers (edges).
--- **Speed:** Exceptional, especially as data grows. Traversal time is proportional to the size of the *traversed path*, not the total database size.
--- **Scalability:** Excellent for relationship-heavy data. Adding more data doesn't degrade performance of traversals if the graph remains localized.
-------------------------------------
*Note: The choice depends on the workload. Use MySQL for strict consistency, MongoDB for high-write distributed workloads, and Neo4j for highly connected, relationship-heavy data.*

## Explain sharding. What problems does it solve.
What is the price of using it?
###
What is Sharding?
Sharding is a horizontal scaling technique that partitions data across multiple database servers (shards). Each shard holds a subset of the data, and the system routes queries to the appropriate shard(s).
------------------------------
What Problems Does It Solve?
--- Scalability limits of a single server (CPU, RAM, disk I/O).
--- Handles massive data volumes and high read/write throughput.
--- Improves performance by distributing load.
--- Increases storage capacity and availability (if replicas are used per shard).
-------------------------------
How it Works
--- Data is split using a shard key (e.g., user_id, region). Common strategies: range-based, hash-based, or directory-based.
--- In Your Project Databases:
- MongoDB: Native sharding support with very good tooling.
- PostgreSQL: Manual or with Citus extension.
- Neo4j: Causal Clustering with sharding in Enterprise edition.
--------------------------------
The Price of Using Sharding (Drawbacks)
--- Increased Complexity: Query routing, data distribution, and rebalancing logic.
--- Distributed Transactions: Harder and slower (or limited) across shards.
--- Joins and Aggregations: Often expensive or impossible across shards — requires application-level handling.
--- Operational Overhead: Monitoring, backup, recovery, and schema changes become much harder.
--- Hotspots: Poor shard key choice leads to uneven load.
--- Higher Costs: More servers, network traffic, and management effort.
--- Eventual Consistency risks in some setups.
--- Recommendation: Shard only when you actually outgrow a single powerful server (or replica set). Prefer vertical scaling and good data modeling first.

## Explain replication. What problems does it solve.
What is the price of using it?
###
What is Replication?
Replication is the process of copying and maintaining database data across multiple servers (replicas). Changes made on a primary node are automatically propagated to secondary nodes.
-------------------------------
What Problems Does It Solve?
--- High Availability & Fault Tolerance: If the primary server fails, a replica can take over (failover).
--- Read Scalability: Offload read queries to replicas, reducing load on the primary.
--- Disaster Recovery: Replicas act as live backups in different locations/zones.
--- Data Locality: Place replicas closer to users for lower latency.
--- Backup without performance impact: Backups can run on replicas.
-------------------------------
Types:
--- Synchronous: Strong consistency, but slower.
--- Asynchronous: Better performance, but possible replication lag.
-------------------------------
The Price of Using Replication (Drawbacks)
--- Consistency Lag: Replicas may not have the latest data (especially async).
--- Increased Complexity: Managing failover, conflict resolution (in multi-master), and monitoring.
--- Higher Resource Costs: More servers, storage, and network bandwidth.
--- Write Overhead: Every write is duplicated across replicas.
--- Potential for Split-Brain scenarios in certain failures.
--- Recommendation: Use replication for production systems. Combine with sharding for extreme scale. This is a fundamental topic for your oral exam.

## Explain partitioning. How is it different from sharding?
###
Partitioning
Partitioning (also called table partitioning) splits a large table into smaller, logical sub-tables (partitions) within the same database instance.
The database engine manages partitions transparently.
------------------------------
Main Benefits:
--- Faster queries (partition pruning — only relevant partitions are scanned).
--- Easier maintenance (drop old partitions, rebuild indexes faster).
--- Better data management for time-series or large datasets.
--- Improved performance on very large tables.
--- Types: Range, List, Hash partitioning.
--------------------------------
Sharding
Sharding distributes data across multiple separate database servers (shards). Each shard is an independent database instance.
Key Difference:
--- Partitioning: Same server, same database engine. Transparent to the application.
--- Sharding: Multiple servers. Requires a shard key and routing logic (often handled by the database or application).
------------------------------------
Summary:
Use partitioning when one powerful server is not enough for a single table’s performance/maintenance.
Use sharding when you need to scale beyond one server’s capacity (horizontal scaling).
--------------------------------------
In Your Project:
--- SQL (PostgreSQL/MySQL): Excellent built-in partitioning support.
--- MongoDB: Combines both (sharding + collections can be partitioned).
--- Neo4j: Uses partitioning internally in clusters.
--- Partitioning is easier and has lower complexity than sharding.

## Explain vertical vs. horizontal scaling of databases.
###
Vertical Scaling (Scaling Up)
Increasing the resources (CPU, RAM, storage) of a single database server.
Example: Upgrading from a 16-core/64GB machine to a 64-core/512GB machine.
Advantages:
--- Simpler to implement and manage.
--- No changes to application code.
--- Better performance for complex joins and transactions.
--- Easier consistency (single instance).
------------------------------------
Disadvantages:
--- Hardware limits (maximum server size/cost).
--- Single point of failure.
--- Downtime possible during upgrades.
--- Expensive at very high scale.
------------------------------------
Horizontal Scaling (Scaling Out)
Adding more database servers and distributing the workload across them (sharding + replication).
Advantages:
--- Virtually unlimited scale.
--- Better fault tolerance and high availability.
--- Cost-effective (use commodity hardware/cloud instances).
--- Handles massive read/write throughput.
------------------------------------
Disadvantages:
--- Increased complexity (data distribution, routing, consistency).
--- Harder joins and distributed transactions.
--- Application changes often required.
--- Operational overhead (monitoring, rebalancing).

#####
Suitability: Horizontal vs Vertical Scaling
Relational Databases (SQL)
--- Vertical Scaling: Excellent. Strong single-node performance for complex joins, transactions, and ACID guarantees. Easy to scale up hardware.
--- Horizontal Scaling: Moderate / Challenging. Possible via read replicas, partitioning, or extensions (e.g., Citus for PostgreSQL). Distributed transactions and cross-shard joins are expensive and complex. Best for moderate scale.
--------------------------------
MongoDB (Document DB)
--- Vertical Scaling: Good for smaller workloads, but limited by single-node capacity.

## CAP Theorem.
###
--- CAP Theorem (Eric Brewer, 2000) states that in a distributed data store, it is impossible to simultaneously guarantee all three of the following properties:
--- Consistency (C): Every read receives the most recent write or an error. All nodes see the same data at the same time.
--- Availability (A): Every request receives a (non-error) response, without guarantee that it contains the most recent write.
--- Partition Tolerance (P): The system continues to operate despite network partitions (communication failures between nodes).
--- Key Insight
You can only choose two out of the three in a distributed system.
#### questionlist distributed
#####
The Three Combinations
CP (Consistency + Partition Tolerance)
Sacrifices Availability.
System may become unavailable during network partitions to maintain consistency.
Examples: Traditional RDBMS in distributed mode, Neo4j Causal Cluster (strong consistency focus), MongoDB with strong write concerns.
-------------------------------
AP (Availability + Partition Tolerance)
Sacrifices strong Consistency (eventual consistency).
System remains available but may return stale data.
Examples: Cassandra, DynamoDB, many MongoDB configurations.
----------------------------------
CA (Consistency + Availability)
Sacrifices Partition Tolerance.
Only possible in non-distributed (single-node) systems.
Not realistic for modern distributed databases.
------------------------------
--- Relational (SQL): Often prioritizes C (strong ACID) → harder horizontal scaling.
--- MongoDB: Flexible — can tune toward CP or AP depending on configuration.
--- Neo4j: Strong emphasis on Consistency + Partition Tolerance in clusters.
--- Modern Note: Most systems aim for Eventual Consistency and use techniques
like CRDTs or quorum reads/writes to balance the trade-offs.





## Consistency vs. availability trade-off.
###
Consistency vs. Availability Trade-off
This is one of the fundamental trade-offs in distributed systems, directly related to the CAP Theorem.
-------------------------
Consistency
--- Every read returns the most recent write (or an error).
--- All nodes see the same data at the same time.
--- Strong consistency = ACID-style guarantees.
--- Pros: Predictable, correct results. Critical for banking, inventory, bookings.
--- Cons: Higher latency, lower availability during network issues or high load.
--------------------------
Availability
--- Every request gets a response (even if it might be stale).
--- The system remains responsive even during failures or partitions.
--- Pros: High throughput, low latency, better user experience.
--- Cons: Risk of stale or conflicting data (eventual consistency).
---------------------------
The Trade-off
--- In distributed databases, you often cannot have both strong consistency and full availability during network partitions.
You must choose:
--- Strong Consistency (CP systems) → May return errors or delay responses during partitions.
--- High Availability (AP systems) → Returns responses quickly, but data may be eventually consistent.
#### questionlist distributed
#####
Real-World Application in Your Databases
--- Relational (SQL): Usually prioritizes Consistency (strong ACID). Better vertical scaling.
--- MongoDB: Tunable — can choose strong consistency (majority write concern) or higher availability (eventual consistency).
--- Neo4j: Strong consistency focus in Causal Clustering.
--------------------------------------
Best Practice:
Choose based on business needs. Use strong consistency for critical data (money, orders)
and eventual consistency for non-critical data (caches, analytics, user feeds).
This trade-off is essential for understanding modern database design and your project’s migration decisions.






## Example of eventual consistency.
###
Eventual Consistency means that if no new updates are made to a data item,
eventually all accesses will return the last updated value.
The system becomes consistent “eventually,” but may return stale data temporarily.
------------------------------
Classic Real-World Example: Social Media “Like” Counter
--- User A likes a post → write goes to one replica (Node 1).
--- User B reads the same post a second later → might still see the old count because the update hasn’t propagated to Node 2 yet.
--- After a short delay (milliseconds to seconds), the replication process updates Node 2.
--- Eventually, both users see the same correct like count.
-------------------------------
Other Common Examples
--- DNS propagation after changing a domain record.
--- Shopping cart in a large e-commerce site (temporary inconsistency between services).
--- User profile updates across multiple regions.
--------------------------------
In Your Project Databases
--- MongoDB: Default behavior in replica sets is eventual consistency for secondary reads. You can tune it with readConcern and writeConcern.
--- Neo4j: Usually strong consistency, but can show eventual behavior in very large clusters.
--- Relational: Rarely uses eventual consistency (prefers strong consistency).
--------------------------------
Trade-off:
--- Gives high availability and low latency, but requires careful application design (e.g., “refresh to see latest” messages) for cases where temporary staleness is acceptable.
--- This example clearly illustrates the Consistency vs Availability trade-off from CAP Theorem.

## DTO pattern
###
DTO Pattern (Data Transfer Object)
What is DTO?
A DTO is a simple, plain object used to transfer data between layers of an application (e.g., from database layer to service layer to controller/API).
It contains only data (no business logic) and is optimized for specific use cases.
----------------------------------
Why Use DTOs?
Advantages:
--- Decouples layers: Hides internal entity structure from the outside world.
--- Security: Prevents over-exposure of sensitive fields (e.g., passwords, internal IDs).
--- Performance: Transfer only needed fields (avoid sending entire entity with relationships).
--- Flexibility: Different DTOs for different clients (mobile vs web, admin vs user).
--- Versioning: Easier to evolve APIs without breaking internal models.
--- Reduces network payload and serialization cost.
----------------------------------
Disadvantages:
Extra boilerplate (many classes).
Mapping effort (Entity ↔ DTO) — usually solved with MapStruct, ModelMapper, etc.
---------------------------------
"""
// Entity (Database layer)
@Entity
public class User { ... full model with relations }
// DTO (API layer)
public class UserResponseDTO {
    private Long id;
    private String name;
    private String email;
    // no password, no internal fields
}
"""

## Transaction boundaries and implementation in the code.
###
Transaction Boundaries in the Arimaa-Backend Project:
The project utilizes Spring's declarative transaction management via the @Transactional annotation. This approach provides a robust and clean way to manage transaction boundaries at the service layer.
---------------------
Implementation:
--- The @Transactional annotation is applied directly to service implementation methods (e.g., in PlayerServiceImpl, MatchServiceImpl, UserServiceImpl).
--- By default, these transactions are read-write.
--- For methods that only perform read operations, @Transactional(readOnly = true) is used to optimize performance and prevent unnecessary write locks.

## N+1 query problem and solutions: incl.
1: Lazy vs Eager loading configuration.
2: Model vs Query configuration.
###
What is the N+1 Query Problem?
It occurs when an ORM loads a collection of entities (1 query), then executes N additional queries to load related data (e.g., for each entity). This leads to severe performance degradation.
Example: Loading 100 users + their orders → 1 + 100 = 101 queries.
--------------------------
1. Lazy vs Eager Loading Configuration
--- Lazy Loading (Default in most ORMs):
Related data is loaded only when accessed.
Pros: Saves memory and initial query cost.
Cons: Causes N+1 if accessed in a loop.
Config (JPA/Hibernate): @ManyToOne(fetch = FetchType.LAZY)
--- Eager Loading:
Related data is loaded immediately in the same query.
Pros: Avoids N+1.
Cons: Can load too much unnecessary data (cartesian product risk).
Config: @ManyToOne(fetch = FetchType.EAGER)
----------------------------------
2. Model vs Query Configuration (Better Solutions)
--- Model Level (Entity annotations): Use FetchType.EAGER sparingly. Risky for global behavior.
--- Query Level (Recommended):
Use JOIN FETCH in JPQL:
"""
SELECT u FROM User u JOIN FETCH u.orders
"""
--- Hibernate: EntityGraph or @EntityGraph
--- Batch fetching: hibernate.default_batch_fetch_size
--- Use DTO projections or native queries for complex cases.
#### questionlist datalayer
#####
Best Solutions Summary:
--- JOIN FETCH / EntityGraph (most effective)
--- Batch fetching
--- Convert to DTOs with custom queries
--- Use native SQL/Cypher when needed
--- Enable SQL logging and EXPLAIN to detect N+1 early
------------------------
In Your Project: Critical when migrating from SQL (ORM) to MongoDB/Neo4j — N+1 issues change form (e.g., many small document lookups).
This is a very common performance question in exams.






## Connection pooling
###
What is it?
Connection pooling is a mechanism that maintains a reusable cache (pool) of open database connections. The application borrows a connection from the pool, uses it, and returns it instead of opening/closing a new connection for every operation.
-------------------------------
Why Use It?
--- Creating a new database connection is expensive (network handshake, authentication, memory allocation).
--- Reduces latency and CPU overhead.
--- Prevents the database from being overwhelmed by too many simultaneous connections.
--------------------------------
How It Works
--- Pool creates a set of connections at startup (or on demand).
--- Application requests a connection → pool provides one (or creates if below max limit).
--- After query execution, connection is returned to the pool.
--- Idle connections can be validated or closed automatically.
---------------------------------
Popular Implementations
--- Java: HikariCP (best performance), Tomcat JDBC, DBCP.
--- MongoDB Driver: Built-in pooling (configurable via MongoClientSettings).
--- Neo4j Driver: Built-in connection pool per driver instance.
--------------------------------------
Key Configuration Parameters
--- maximumPoolSize (e.g., 10–50) – max concurrent connections.
--- minimumIdle – keep some connections ready.
--- connectionTimeout – wait time before failing.
--- maxLifetime / idleTimeout – recycle old connections.
----------------------------------------
--- Benefits: High throughput, better scalability, lower resource usage.
--- Drawbacks: Misconfiguration can lead to connection leaks or exhausted pools.
--- In Your Project:
Use HikariCP for the SQL part with Spring Boot. Tune pools separately for each database (SQL, MongoDB, Neo4j) based on expected load. Monitor metrics like active connections and wait times.

## Data Consistency at application level: incl.
1: Idempotency.
2: Optimistic vs. Pesimistic locking
3: Row versioning.
###
1. Idempotency
--- Idempotency ensures that repeating the same operation multiple times has the same effect as doing it once.
--- Why important?
Prevents duplicate effects from network retries, timeouts, or duplicate requests.
---------------------------
Implementation:
--- Use unique request IDs or idempotency keys (stored in DB).
--- Make operations like CREATE ORDER idempotent by checking if the key already exists.
--- HTTP methods: PUT and DELETE are naturally idempotent; POST usually isn’t.
------------------------
2. Optimistic vs Pessimistic Locking
Pessimistic Locking
--- Locks the record immediately on read (e.g., SELECT ... FOR UPDATE).
--- Pros: Prevents conflicts early.
--- Cons: Reduces concurrency, risk of deadlocks, bad for long operations.
--- Best for: High-contention scenarios with short transactions.
-----------------------------
Optimistic Locking
--- No lock on read. Check for changes before write (usually via version column).
--- Pros: High concurrency, better scalability.
--- Cons: Conflicts detected late → requires retry logic.
--- Best for: Low-contention, read-heavy workloads.
#### questionlist datalayer
#####
3. Row Versioning
--- Add a version column (integer or timestamp) to entities.
--- How it works:
- On read: Load current version.
- On update: UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?
- If 0 rows affected → conflict (another transaction modified the row).
--------------------------
--- Benefits: Simple implementation of optimistic locking. Works well with ORMs (JPA @Version).
--- In Your Project:
- Use optimistic locking + versioning in the SQL part. Handle retries in the application layer when migrating data to MongoDB and Neo4j (which have their own concurrency models).
- This ensures application-level consistency even when database transactions are not sufficient.

## auditing: what is auditing used for?
###
Auditing is the systematic recording and monitoring of database activities and changes.
--------------------
Main Purposes:
--- Security & Compliance
Track who accessed or modified sensitive data (GDPR, HIPAA, PCI-DSS, SOX compliance).
Detect unauthorized access or insider threats.

Essential for regulatory audits and legal investigations.
--- Forensic Analysis
Investigate security incidents, data breaches, or errors.
Reconstruct what happened (who changed what and when).
--- Troubleshooting & Monitoring
Identify performance issues (slow queries, unusual patterns).
Monitor schema changes, privilege escalations, and configuration modifications.
--- Operational Control
Track backup/restore activities.
Monitor data quality and business rule violations.
----------------------------
What is typically audited?
--- Login/logout and failed login attempts
--- DDL (CREATE, ALTER, DROP)
--- DML on sensitive tables (INSERT, UPDATE, DELETE)
--- Privilege changes (GRANT, REVOKE)
--- Access to critical data

Breakdown:
DDL = Data Definition Language
These commands change the structure of the database:

## auditing: Performance impact of auditing.
###
Performance Impact of Auditing
Auditing adds overhead because every tracked operation generates extra work.
------------------
Main Performance Costs:
--- Increased I/O: Every audited DML (INSERT/UPDATE/DELETE) causes additional writes to audit tables or logs.
--- CPU Overhead: Triggers or application logic must capture old/new values, format JSON, and record metadata.
--- Lock Contention: Triggers can extend transaction duration, increasing lock hold time.
--- Storage Growth: Audit tables grow rapidly and can become very large if not partitioned or purged.
--- Query Impact: Large audit tables can slow down the entire database if not properly indexed or maintained.
-----------------------
When the Impact is Highest:
--- High-frequency write tables (orders, transactions, user activity).
--- Auditing every column with full old/new values.
--- Using row-level triggers on busy tables.
------------------------
Mitigation Strategies:
--- Audit only sensitive/critical tables and columns.
--- Use asynchronous auditing (write to queue → background process).
--- Partition audit tables by time (changed_at).
--- Limit auditing to UPDATE/DELETE on important data (skip routine INSERT).
--- Use efficient storage (JSONB in PostgreSQL).
--- Monitor with slow query logs and pg_stat_statements.
----------------------------
Recommendation for your project:
--- Enable auditing on key tables in the SQL database, but keep it lightweight. In MongoDB and Neo4j, use built-in change streams / logs selectively.
--- Balance: Auditing is essential for security and compliance, but over-auditing can noticeably degrade performance. Always measure impact with realistic load.

## dbmodelling: ERD:

1. Standard Diagram Notations
--- Entities: Represented as rectangles (e.g., User, Order).

--- N:M (Many-to-Many) – usually resolved with a join table

- Mandatory (Total Participation): Shown as a straight line or filled circle. The entity must participate in the relationship.
- Optional (Partial Participation): Shown as a circle (O) or dashed line. The entity can exist without participating.

## Normalization:
1: Explain first 3 normal forms without using formal definitions.
2: What problems does normalization prevent?
###
Normalization: First 3 Normal Forms (Simple Explanation)
--- 1NF (First Normal Form)
Each column contains only atomic (single) values. No repeating groups or multiple values in one cell. Every row must be unique.
--- 2NF (Second Normal Form)
The table is in 1NF, and every non-key column depends on the whole primary key, not just part of it. Removes partial dependencies.
--- 3NF (Third Normal Form)
The table is in 2NF, and no non-key column depends on another non-key column (no transitive dependencies). Every column depends only on the primary key.
-----------------------------
What Problems Does Normalization Prevent?
--- Update Anomalies — Changing one fact requires updating many rows (risk of inconsistency).
--- Insertion Anomalies — Cannot insert certain data without creating fake/dummy records.
--- Deletion Anomalies — Deleting a record removes unrelated valid information.
--- Redundancy — Saves storage and reduces inconsistency risk.

## Referential integrity: incl:
1: How is it enforced in RDBMS?
2: How is it with referential integrity in document and graph databases?
3: What is an orphaned record?
###
1. How is it enforced in RDBMS?
In relational databases, referential integrity is enforced using Foreign Key (FK) constraints.
--- The FK column must reference a valid Primary Key (or Unique) value in the parent table.
--- The database automatically prevents invalid inserts/updates and can handle deletes via rules like:
"""
ON DELETE CASCADE
ON DELETE RESTRICT / NO ACTION
ON DELETE SET NULL
"""
-----------------------------
2. Referential Integrity in Document and Graph Databases
--- MongoDB (Document DB): No automatic enforcement.
--- You must handle it in the application code. Common approaches: embedding data instead of referencing, or manual checks + schema validation.
--- Neo4j (Graph DB): Stronger than MongoDB.
--- Relationships are first-class objects. You cannot have a relationship pointing to a non-existent node. Use DETACH DELETE when removing nodes.
-------------------------
3. What is an Orphaned Record?
--- An orphaned record is a child record that references a parent record that no longer exists (a dangling reference).
--- Example: An Order that still points to a deleted Customer.
--- Key Point for your project: SQL provides strong automatic protection,
while MongoDB and Neo4j shift more responsibility to the application layer.

## How to handle existing production data during the schema migration?
###
Plan Carefully
--- Use version-controlled migration tools (Flyway, Liquibase for SQL).
--- Always test migrations on a copy of production data first.
-----------------------
Common Strategies:
--- Additive Changes (Safest): Add new columns/tables with default values or NULL. Old data remains valid.
--- Backfill Data: Run scripts to populate new fields based on existing data (e.g., set default values, calculate new fields).
--- Data Transformation: Use scripts or ETL tools to transform old data into the new schema.
--- Dual Write: Temporarily write to both old and new structures during transition.
--- Blue-Green Deployment: Run new schema in parallel and switch traffic after validation.
------------------------
For Your Three Databases:
--- SQL: Use migrations with ALTER TABLE + backfill scripts. Maintain backward compatibility.
--- MongoDB: Much easier — just start writing new document structure. Use aggregation pipelines or scripts for data migration.
--- Neo4j: Add new nodes/relationships. Use Cypher scripts to migrate existing data.
-----------------------
Key Rules:
--- Never lose data (always backup first).

## Data migration cost?
###
Main Cost Factors:
--- Time & Development Effort: Designing target models, writing ETL/migration scripts, data transformation logic, and handling edge cases.
--- Performance & Downtime: Large datasets take hours/days to migrate; may require downtime or complex zero-downtime strategies.
--- Testing & Validation: Ensuring data integrity, completeness, consistency, and correctness across all three databases (very time-consuming).
--- Infrastructure: Temporary storage, extra compute resources, and tools for migration.
--- Application Changes: Updating code, queries, and business logic for the new data models.
--- Risk & Rework: Data loss, inconsistencies, or bugs can lead to expensive fixes.
--------------------------
Summary:
--- Data migration is often one of the most expensive and risky parts of a project. SQL → MongoDB (denormalization) and SQL → Neo4j (graph modeling) transformations add significant complexity and cost.
--- Best Practice for your exam project:
Migrate in small batches, automate as much as possible, and thoroughly test data consistency. Always calculate migration cost when choosing architectures.

## Schema migration in NoSQL databases?
###
MongoDB (Document DB)
--- Very flexible due to schema-less nature.
--- You can insert documents with new fields without changing any schema.
--- Old and new document versions can coexist in the same collection.
--- Migration techniques:
- Background scripts using updateMany() or aggregation pipelines.
- Add a version field to documents for compatibility.
- Use JSON Schema Validation (optional) to gradually enforce structure.
--------------------------------
Neo4j (Graph DB)
--- Highly agile and schema-flexible.
--- Migration techniques:
- Add new properties to existing nodes/relationships on the fly.
- Create new relationship types or node labels.
- Use Cypher scripts (MATCH ... SET ... CREATE ...) to transform data.
--- No structural locking issues.
-------------------------------
Advantages vs SQL:
--- Much faster, lower risk, and minimal downtime.
--- No need for complex migration tools like Flyway/Liquibase.
--- Easier to evolve schema as requirements change.
----------------------------------
Best Practice:
Always migrate data in small batches, keep backward compatibility, and thoroughly validate after changes.
This flexibility is one of the biggest advantages of NoSQL in your project.

## How to migrate schema and production data between database technologies?
###
General Strategy (SQL → MongoDB → Neo4j)
--- Extract from Source (SQL)
- Use pg_dump, mysqldump, or custom SELECT queries.
- Export in structured format (CSV, JSON).
------------------------
Transform
--- Map relational model to target model:
- MongoDB: Denormalize → Embed related data (e.g., orders inside user document).
- Neo4j: Convert tables to nodes, foreign keys to relationships.
--- Handle data cleaning, type conversion, and enrichment.
------------------------------
Load into Target
--- MongoDB: Use mongoimport, custom scripts, or MongoDB drivers with bulk inserts.
--- Neo4j: Use LOAD CSV, Cypher scripts, or APOC procedures for batch import.
