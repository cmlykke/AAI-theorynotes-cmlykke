## What is SDLC and what is the V model
### SDLC stands for Software Development Life Cycle.
Requirements -> Design -> Development -> Testing -> Deployment -> Maintenance.
#### chrnoter
##### there are several different SDLC models:
(Waterfall, Agile, Spiral, Iterative, DevOps, etc.). The V-model is one of them.
######
The V-Model (Verification and Validation Model) is a sequential SDLC model that gives strong emphasis on testing. It is an extension of the classic Waterfall model, but instead of testing being only at the end, testing activities are planned in parallel with each development phase.
It gets its name from the "V" shape of the diagram:
** Left side of the V → Development (downward flow)
** Bottom → Coding
** Right side of the V → Testing (upward flow)
----
Visual Structure of the V-Model:
1: Requirements Analysis          ↔          Acceptance Testing
       ↓                                           ↑
2: System Design                  ↔          System Testing
       ↓                                           ↑
3: Architectural (High-Level) Design ↔     Integration Testing
       ↓                                           ↑
4: Module (Low-Level) Design      ↔          Unit Testing
       ↓                                           ↑
5:             CODING (Implementation)

## black box - Equivalence partitioning
### Instead of testing every possible input value (which is usually impossible), you divide the input domain into a small number of equivalence classes (or partitions).
A partition: "A partition (also called an equivalence partition) is a set of input values that the software is expected to treat in exactly the same way."
Valid partitions → Inputs that should be accepted by the system.
Invalid partitions → Inputs that should be rejected or handled with error messages.
#### chrnoter blackbox
#####
For integers: positive tal, 0 og negative tal er partitions.
hvis systemet kun acceptere en range, f.eks. 18-65 for at få kørekort,
vil de positive tal 18-65 være en valid partiton, og 1-17 og 66+ vil være invalid partitions.
Herudover vil 0, negativetal, floats, strings, null ect. være invalid partitions.
Hvis man arbejder i et typestærkt sprog, kan non-integers udelades som partitions, fordi systemet ikke vil acceptere dem.
######
For strings:
empty string (iflg. Arturo. Korikov gør det ikke), only whitespace, minimul / maximum length,
valid / invalid characters (eg. ø, ö, 漢).
(ISTQB (International Software Testing Qualifications Board) explicitly mention prioritizing invalid partitions that represent security risks, such as SQL injection attempts.)
For dates: minimum / maximum value, invalid format, leap-year (2024-02-29 valid, 2025-02-29 invalid - and always test for 30.feb.), timezone.



## blackbox - Boundary value analysis
### "errors tend to cluster around boundaries".
We use 3-value boundaries: (partition: 1-99. Valid boundaries: 1, 2, 98, 99, invalid boundaries: 0, 100). Man kan vælge også at medtage en "Middle value". For partition 1-99: 23 could be a middle value.
(Det er valgfrit at bruge middle-values, så længe man har
gode 3-value boundaries)
For overlapping values: remove dublicates.
#### chrnoter blackbox
##### Treat type-specific extremes as boundaries:
example: signed 32-bit integers:
INT_MIN (-2147483648), INT_MIN + 1, -1, 0
INT_MAX (2147483647), INT_MAX - 1, 1.
(Dette kan vi ikke gøre for strings, så vi må beslutte for store strings vi vil teste).
######
Når man tester ekstreme værdier (eg. MAX_INT). Hvordan tester man så MAX_INT + 1?:
"because of wraparound, you test the extreme representable values and the system's behavior when arithmetic crosses those limits — not necessarily by passing the wrapped result as input."-AI

Dog bør man altid undersøge hvad appen skal bruges til.

## black-box: TDD: London vs. classical
###
Classical (Detroit/Chicago) (State-based) School:
-- Verification style: Check final state or return value.
-- Use of Test Doubles: Minimal – prefer real objects
-- Isolation: (Kent Beck TDD book): Tests can run with real dependencies.
Mock only shared dependencies (collaborators (shared dependencies) are not mocked)
-- Design Influence: Emergent design from refactoring
-- fordel: det er nemere at refakturere, pga. focus på returnerede værdier. Fravær af mocks beskytter også bedre mod regression.
-- ulempe: sværere at finde årsagen til bugs, og testene vil ofte køre langsommere, når få ting er mocked.
#### chrnoter blackbox ainotes
#####
London / Interaction-based school:
-- Verification style: Check interactions with collaborators
-- Use of Test Doubles: Heavy use of mocks/stubs/spies
-- Isolation: (Steve freeman & Nat Price 2009): Isolate SUT from all dependencies. (but dont mock constants / enums ect.)
-- Design Influence: Design guided by interfaces & interactions
-- fordel: På grund af isolering, ved man præcist hvad der fejler, og kan hurtigt fixe det.
-- ulempe: når man isolere alle systemets dele, kræves der mange tests, og al refakturering / kodeændring kræver meget arbejde med at ændre tests
######
"the London School vs Classical School distinction originated in and is most strongly associated with Test-Driven Development (TDD), but the core ideas extend beyond TDD to general unit testing practices. It was popularized by Martin Fowler in his 2007 article "Mocks Aren't Stubs"." - AI
--- Khorikovs aproach: classical + “dependencies we have full occntrol over, eg. databases, should NOT be mocke”.


## black-box: TDD: Mock, Stub and Spy
### Alle 3 kaldes "test doubles".
Man mocker de objekter der kræver for lant tid (testene bliver for langsomme at køre).
I Jest (javascript) kan man ogsaa mocke implementationen af en funktion, f.eks. Hvis den er kompliceret og man kun er interesseret I en lille del.
#### chrnoter blackbox
#####
London school:
-- Mock: Outgoing information (eg. send an email). (verify that system under test sent the correct data)
-- Stub: Incomming information (getting infor from database) == Stub. (provides fake data to the system under test)
-- Spy == Does both.
Classical school:
All 3 are mocks.
######
A "Spy" is a test double that acts as both a Stub and a Mock at the same time. (it is usually a wrapper around a real object)
Uses:
1: You want most of the real object’s behavior, but need to override just one or two methods.
2: testing legacy code: When you can’t easily refactor a big class, but still want to isolate parts of it.
3: Verifying Side Effects while keeping real logic.
4: observe internal calls: You need to know whether private/protected/helper methods inside the class were invoked correctly.


## black-box: what to test in a unit test
###
-- Unit tests should test one behavior
-- en behavior, can vaere en funktion, men det kan ogsaa vaere meget mere komplex.
-- AVOID chaining arrange, act assert.
-- Avoid branching (if else while for)
-- Cover the business logic, and not much else.
(DET BETYDER == I MANGE APPLICATIONER BOER DER IKKE VAERE UNIT TESTS !!! MEDMINDRE MAN BRUGER TDD)
-- "Act" section should be one line (eg. hvis man har en Purchase funktion, og er noedt til bagefter at kalde en RemoveFromInventory, betyder det at koden er dårlig.RemoveFromInventory burde have vaeret en del af  Purchase handling)
-- Never test private methods / functions directly. (hvis man gør det, kopler man tests til implementation)
-- Do never expose private state (Do not leak domain knowledge to unit tests -> use hardcoded or mocked values in tests)
-- dont polute the code with anything test-related (eg. dont use "only for test"-booleans)
#### chrnoter blackbox
##### Årsagen til at man kun bør teste businesslogic:
jo mere man tester ting udover business logic, jo mere er man noedt til at aendre dem naar man refakturere. Der boer ikke ske. Unit tests skal vaere implementationsuafhaengige.

-- Continous delivery: After CI succeeds, the pipeline goes further: It automatically prepares the software for real-world use and runs more thorough tests (integration test, enc-to-end ect). The final step to production needs human approval.

## Review (1)
###
two types:
1: Codereview
2: Document review
2a: document is studied by the reviewers
2b: Author is informed of the issues found
2c: the author updates the document
2d: a next iteration is performed, if nessasary
#### chrnoter review
#####
Review: Systematic examination of a document to find and remove errors.
(objective: Finding defects, gaining understanding, generating discussions, decision making by consensus.)
--- Reviews in the V-model:
We can review anything in the V-model.
######
4 types of document reviews (by levels of formality):
1: informal
1a: Documentation not required
1b: Sometimes just a quick non-technical check
1c: inexpensive, but limited benefit
2: Walkthrough (can be code review)
2a: Review meeting led by the author of the document
2b: It helps team members understand the document
2c: open-ended
2d: preparation and production of a repost is optional
3: Technical review (code review)
3a: Documented, well-defined detection process.
3b: peer review, desirably with no presence of management.
3c: Preparation and report.
4: Inspection
4a: led by a trained moderator.
4b: Peer examination of a document.
4c: pre-meeting preparation
4d: production of an inspection report
4e: formal follow-up
